# 한국 공공/금융 제안요청서(RFP) 골격 — 요구사항 15유형 · 7항목 작성표 · 평가배점

> deep-research 검증 결과(3-0 confirmed, primary source). 한국 제안요청서의 **과업내용/요구사항정의서/평가** 골격.
> ★ 본 문서의 모든 수치·prefix는 1차 출처 검증본. **반증된 변형은 [research-provenance.md](research-provenance.md)에 격리** — 절대 주장 금지.

## §1. 요구사항 15유형 분류 (NIPA/SWIT 요구사항정의 방법론)

공공 SW사업 요구사항은 **15개 유형**으로 표준 분류 — 기능(1) + 비기능(10) + 기타(4). 각 유형은 **3-letter 영문 ID prefix**로 고유번호 부여(예 `SFR-001`).
출처: NIPA/SWIT 「공공SW사업 제안요청서 작성을 위한 요구사항 상세화 실무가이드라인」(2018) 원문 verbatim — *"현재 소프트웨어사업에서 도출될 수 있는 요구사항은 15개 유형으로 정의되며, 기능(1개)과 비기능(10개), 기타(4개) 요구사항으로 구분"*.

| 분류 | ID | 한글명 | 적용 예(일반) |
|---|---|---|---|
| **기능(1)** | **SFR** | 기능 요구사항 | 화면·카드 resolver, 지표 판정/색상, drill-down, 즐겨찾기, AI 에이전트 Q&A 등 |
| 비기능(10) | **PER** | 성능 | cache TTL/jitter, Single-Flight, FE 동시성 상한, 질의 latency 목표 |
| | **ECR** | 시스템장비구성 | N-tier(FE/BE/AI) + DBMS + 캐시(예: Sentinel) + 모델 런타임 |
| | **INR** | 인터페이스 | 외부 연계 N종(보안필터 / 도메인 API / 검색엔진 등), BE↔AI |
| | **DAR** | 데이터 | DB 스키마, 마스터 차원 규모, 적재·리시드 파이프라인 |
| | **TER** | 테스트 | 골든 verify(빌드/유닛/통합), 시각회귀, e2e |
| | **SER** | 보안 | 입출력 검증, prompt injection 방어, CVE/SBOM 스캔, 시크릿 비노출 |
| | **QUR** | 품질 | SoT 1:1 정합, LLM 사실정확도/환각률, 접근성 |
| | **COR** | 제약사항 | 기술스택 고정, 금지 패턴(예: ORM/migration/SSR), SQL 외부화 등 |
| | **PMR** | 프로젝트관리 | 단계·마일스톤, 과업심의, 진척보고 |
| | **PSR** | 프로젝트지원 | 인력투입, 형상관리, 산출물 관리 |
| **기타(4)** | **MPR** | 유지관리수행 | 하자보수, 운영이관 |
| | **MHR** | 유지관리인력 | 유지보수 인력 등급/투입 |
| | **CNR** | 컨설팅 | (해당 시) |
| | **ENR** | 공사 | (해당 없음 — SW 외주) |

- ★ **인터페이스 = `INR`** (`SIR` 아님). 일부 2차 출처가 `SIR`로 오기했으나 1차 가이드 원문은 `INR-000` — 반증 1-2(provenance §반증 참조).
- ★ **`SFR` 중의어 주의**: 여기서는 **System Function Requirement(기능)**. Common Criteria의 `SFR`(Security Functional Requirement)와 **다른 개념** — 혼동 금지. 보안은 별도 `SER`.

## §2. 요구사항 내용 작성표 — 7항목 (유형별 1행씩)

각 요구사항은 다음 **7항목 표**로 기술(1차 출처 verbatim):

| # | 항목 | 작성 내용 |
|---|---|---|
| ① | 요구사항 분류 | 15유형 중 하나 (예 SFR) |
| ② | 요구사항 고유번호 | `<prefix>-<일련>` (예 `SFR-001`) |
| ③ | 요구사항 명칭 | 짧은 제목 |
| ④ | 요구사항 상세설명 | 정의 + 세부내용 (testable하게) |
| ⑤ | 산출정보 | 이 요구로 산출될 화면/문서/데이터 |
| ⑥ | 관련 요구사항 | 연관 ID 교차참조 |
| ⑦ | 요구사항 출처 | 근거(spec `file:LINE`, 디자인 `<DesignRef(page-section)>`, 인터뷰 등) |

> 동일재구현 RFP에서는 ⑦ **출처**에 spec `file:LINE` 또는 디자인 인용키 `<DesignRef(page-section)>`를 직접 박아 SoT 추적성을 강제.

## §3. 제안서 평가배점 (조달청 「협상에 의한 계약 제안서평가 세부기준」)

검증본(91p 공식 PDF verbatim, 최신 개정 시행 2025-07-01):

- **기술능력평가 80점 + 입찰가격평가 20점 = 100점.**
- **기술능력평가 = 정성평가 + 정량평가**, 단 **정량평가 총배점한도 ≤ 20점**(*"제3항의 정량평가에 대한 총배점한도는 20점을 초과할 수 없다"*).
- SW/시스템 개발사업 **정성평가 → 별표3의2** (*"소프트웨어 및 시스템 개발사업: 별표3의2"*).
- 정량평가 항목 → 별표8~14: 경영상태 / 수행실적 / 책임성·성실성 / 상생협력 / 유지관리 하도급 적정성 / SW기술자료임치 / 정량필수제안.
- **상위근거**: 기획재정부 계약예규 「협상에 의한 계약체결기준」 + 행안부 예규 「지자체 입찰 낙찰자 결정기준」 제5장. **정보화사업은 「행정기관 및 공공기관 정보시스템 구축·운영 지침」 제18조 기술능력평가 배점한도**(배점한도 90점, HW가 50% 이상이면 80점).

> ★ **반증(주장 금지)**: *"10점 범위에서 분야별 배점한도를 가감할 수 있다"* 는 **0-3 반증**. 위 검증된 구조(80/20, 정량 cap ≤20, 별표 매핑)만 신뢰.
> 금융사 자체발주는 조달청 세부기준을 **그대로 적용할 의무는 아니나**, 협상에 의한 계약의 **사실상 표준 골격**으로 차용 가능(기술 위주 80:20 + 정성/정량 분리).

## §4. 법적 근거 / 방법론 출처 (현행 기준)

- **방법론**: NIPA(정보통신산업진흥원, 現 SW발주기술지원센터) / SWIT 포털 요구사항정의 방법론.
- **현행 법적근거**: 「소프트웨어 진흥법」(전부개정 2020-06-09, **시행 2020-12-10**) — **제44조**(요구사항 명확화·공개) + **제50조**(과업심의위원회). + 「소프트웨어사업 계약 및 관리감독에 관한 지침」 제11조.
- ★ **CURRENCY(필수)**: 구 「소프트웨어산업 진흥법」 제20조제3항(2012 상세요구 공개 의무)은 **전부 폐지·대체**됨. 현행 RFP는 **2020 「소프트웨어 진흥법」**을 인용할 것. (방법론 실체 — 15유형·7항목·80/20 — 는 불변.)
- ★ **반증(주장 금지)**: *"상세화가 구 소프트웨어산업진흥법 제20조③에 따른 법적 의무"* 는 **0-3 반증**(구법 폐지로 근거 무효). 현행 진흥법 기준으로 서술.

## §5. 표준 제안요청서 목차 (위 골격 반영)

1. 사업 개요 (사업명·목적·배경·범위 요약·기간·예산)
2. 현황 및 문제점 (As-Is)
3. **과업 내용** (To-Be) — 과업 범위 + **요구사항정의서**(§1 15유형, §2 7항목표)
4. 제안서 작성 요령 (제안서 목차·규격·분량)
5. **제안서 평가** (§3 배점표 기술80/가격20, 평가절차)
6. 입찰 유의사항 / 계약 조건
7. 산출물 및 추진일정 (마일스톤)
8. 하자보수 / 유지관리 (MPR/MHR)
9. **보안 및 규제 준수 요건** ([compliance-and-ai-llm.md](compliance-and-ai-llm.md) — SER)
10. 별첨 (요구사항 작성표 전체, SoT 인덱스, 제출서식)
