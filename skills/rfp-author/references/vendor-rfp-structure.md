# 요건 발의형 벤더 RFP 구조 (실증 레시피)  ★기본 모드

소프트웨어 프로젝트를 외부 IT 기업에 외주할 때의 **요건 발의형** RFP 구조. "동일재구현"이 아니라 **요건만 발의**(제안사가 충족 방안을 제안).

## 0. 대전제
- **동일재구현 프레이밍 금지**: 제안사가 요건을 충족하는 구현 방안을 제안한다(재현 명령 X).
- **현황(As-Is) 비노출 / 기획(planned) 공개**: PoC 레포 구성·As-Is 섹션은 통째로 삭제. 기획 기능은 요건으로 적극 노출(권위 SoT 최신본 기준).
- **가격·사업기간은 제안받음**: 발주자가 박지 않고 제안사가 제시 → 검토 후 확정.

## 1. 10절 구조
| § | 절 | 핵심 내용 |
|---|---|---|
| 1 | 프로젝트 개요 | 사업명·발주자·배경·목적·기간/예산(제안). "요구사항을 발의"임을 명시 |
| 2 | 요구 기능(Scope) | 2.1 시스템개요 / **2.2 구축규모(물량)** / 2.3 KPI/핵심영역 / 2.4 레이아웃·인터랙션 / 2.5 지표/규칙 / 2.6 데이터 / 2.7 AI에이전트(해당 시) / **2.8 제외범위** |
| 3 | 기술 요구사항 | 스택(호환 요건, 대안 협의)·아키텍처·형상관리 |
| 4 | 보안 요구사항 | 망분리·온프레미스(또는 클라우드 정책)·PIPA 제26조·시큐어코딩·OWASP LLM Top10(해당 시)·원격개발·NDA |

| 6 | 납품 범위 | 소스·DB스키마·테스트(커버리지)·OpenAPI·운영가이드·**OSS 라이선스 보고서** |

## 2. 간이배점 (예시표 차용 → 도메인 조정)
기술 **40** / 보안 **30** / 일정·사업관리·수행인력 **20** / 가격 **10** = **100**.
- 공공 조달청 80/20(정량≤20)은 **직발주 벤더 RFP엔 미사용**(공공 입찰일 때만).
- ★반증 금지: "±10점 분야별 가감"(조달청) 주장 금지 — research-provenance.md §2.

## 3. ★물량(volume) 명시 — 산정 가능성
규모를 안 박으면 수행사가 비용·기간 산정 불가. §2.2에 규모 표를 박고 각 수치에 SoT 근거(쪽번호/파일) 동반:
- 예시 항목군: `<N 화면>` · `<N 카드(=조회 API N+)>` · `<N 지표항목>` · `<N KPI>` · `<마스터 차원 규모>` · `<데이터 종류 N + 합성생성기>` · `<위젯 N종>` · `<AI 카테고리/FAQ/fallback 수, 해당 시>` · `<디자인 N쪽(DesignRef(page-section))>`.
- 전수 상세(인벤토리·카탈로그·시나리오)는 **착수 후 제공 자료**(§10) — 기밀은 격리망/마스킹.

## 4. ★노출 게이트 (벤더 발송본 = 깨끗)
**절대 비포함**(검수 lane grep): 내부 SoT `file:LINE`(`<spec/wiki 경로:NN>`) · 실제 호스트/포트(`<internal-host>`·`<:port>`) · 내부식별자(보안필터 제품명·내부 테이블 prefix·내부 데이터셋명·모델ID) · `.env.*` 값 · "deep-research"/provenance/"추적박스" · em-dash(`—`→`-`). 내부 추적은 본 `references/`에만.

## 5. 쪽번호·인용 의무
디자인/문서 인용 시 **page-NNN/쪽번호 직접 기록**. 디자인 SoT(예: 디자인 PDF/이미지, 인용키 `<DesignRef(page-section)>`)는 페이지 번호를 직접 박는다. "정합 OK" 같은 빈말 금지.
